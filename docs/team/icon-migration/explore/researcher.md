# Exploration: lucide-react to Internal Icons Migration

Date: 2026-04-04

## Research question

What is the current state of the lucide-react to internal icons migration across the `@acronis/shadcn-uikit` monorepo? Specifically: what was the scope of lucide-react usage, what does the internal icon system provide, what has been migrated, and what gaps remain?

## Scope

**In scope:**
- All `lucide-react` imports across `packages/ui`, `packages/demo`, `packages/demos`, `packages/docs`
- The internal icon catalog at `packages/ui/src/components/icons/`
- The `missing-icons.tsx` shim files in `packages/demo` and `packages/demos`
- Re-export and barrel file patterns for icons
- Icon name mapping between lucide names and internal equivalents

**Out of scope:**
- Visual comparison of icon rendering (stroke vs fill style differences)
- Bundle size impact analysis
- Performance benchmarks

---

## Findings

### 1. Migration status: lucide-react has been removed

**Confidence:** Corroborated

The `lucide-react` package is no longer listed as a dependency in any `package.json` across the monorepo. Zero import statements of the form `from 'lucide-react'` exist in any `.ts` or `.tsx` file under `packages/`. The CHANGELOG at `packages/ui/CHANGELOG.md` records two migration commits:

- `b3440f3` — `ui: migrate lucide-react to internal icons`
- `a7935cf` — `ui: migrate lucide-react from stories to internal icons`

All 22 UI component files that previously imported from `lucide-react` now import from `@/components/icons` (20 files) or `@/components/icons/auto-generated` (2 files: `date-picker.tsx`, `sonner.tsx`).

### 3. Internal icon system

**Confidence:** Corroborated

#### 3.1 Architecture

The internal icon system lives at `packages/ui/src/components/icons/` and has this structure:

| Layer | File | Purpose |
|-------|------|---------|
| SVG sources | `svg/` directory | 1,561 SVG files, named `{name}--{size}.svg` (e.g., `chevron-down--16.svg`). Sizes: 16, 24, 32, 72 |
| Base wrapper | `base-icon.tsx` | `BaseIcon` component: wraps SVG children in `<svg>` with `className="size-4 shrink-0"`, default `viewBox="0 0 16 16"`. Also exports `Icon` (accepts `path` prop) |
| Generated components | `auto-generated.tsx` | ~1,148 named exports generated by `npm run generate:icons`. Each wraps an SVG import via `?react` suffix in `BaseIcon` |
| Dynamic loading | `dynamic-icons.tsx` | `getIcon(name)`, `getIconNames()`, `DynamicIcon`, `LazyIcon`, `Icons` record |
| Hand-crafted | `icon-library.tsx` | 7 status icons with hardcoded colors (`InfoIcon`, `SuccessIcon`, `WarningIcon`, `CriticalIcon`, `DangerIcon`, `CloseIcon`, `ChevronDownIcon`). Currently commented out of the public API |
| Categories | `categories.json` | Prefix-based grouping of icon names |
| Public API | `index.ts` | Re-exports `base-icon.tsx`, `dynamic-icons.tsx`, `auto-generated.tsx` |

#### 3.2 Naming convention

Internal icons use **PascalCase with `Icon` suffix**: `ChevronDownIcon`, `SearchIcon`, `CheckIcon`. SVG source files use **kebab-case with size suffix**: `chevron-down--16.svg`, `search--16.svg`.

#### 3.3 Re-exports

The UI package's main `index.ts` at `packages/ui/src/index.ts` exports the icons module. Consumers import via `@acronis-platform/shadcn-uikit` (e.g., `import { SearchIcon } from '@acronis-platform/shadcn-uikit'`). lucide-react was never re-exported from the UI public API.

### 5. The missing-icons shim pattern

**Confidence:** Corroborated

For `packages/demo` and `packages/demos`, icons that have no internal catalog equivalent are provided via a `missing-icons.tsx` file containing inline SVG components with lucide-style stroke paths. Both files are identical (79 exported icon components, 525 lines each).

The shim chain:
- `packages/demos/src/icons/missing-icons.tsx` — canonical source, 79 inline SVG components
- `packages/demo/src/components/icons/missing-icons.tsx` — identical copy
- `packages/docs/src/icons/missing-icons.tsx` — re-exports from `@acronis-platform/shadcn-uikit-demos/icons/missing-icons`
- `packages/demos/package.json` exposes `"./icons/missing-icons"` as a package export

## Key takeaways

- lucide-react is fully removed as a dependency from all `package.json` files across the monorepo. Zero import statements from `lucide-react` exist in any source file. (Corroborated)
- The UI package's 22 component files and 17 story files have been migrated to the internal icon catalog (~1,148 auto-generated components from 1,561 SVGs). (Corroborated)
- 79 lucide icons with no internal equivalent are shimmed via `missing-icons.tsx` files in `packages/demo` and `packages/demos`, containing inline SVG copies of lucide paths (MIT licensed). (Corroborated)
- Two icons (`Moon`, `Sun`) remain as inline SVGs in `packages/ui/src/components/mode-toggle.tsx` with TODO markers, because no `moon--16.svg` or `sun--16.svg` exists in the internal catalog. (Corroborated)
- The `missing-icons.tsx` files are duplicated identically between `packages/demo` and `packages/demos` (525 lines, 79 icons each). `packages/docs` re-exports from `packages/demos`. (Corroborated)

## Open questions

1. **Moon/Sun catalog addition** — The TODO in `mode-toggle.tsx` requests adding `moon--16.svg` and `sun--16.svg` to the internal catalog. What is the timeline and ownership for this?
2. **missing-icons.tsx duplication** — `packages/demo` and `packages/demos` contain identical 525-line files. Should `packages/demo` import from `packages/demos` (like `packages/docs` does) instead of maintaining its own copy?
3. **Stroke vs fill visual consistency** — The 79 missing-icons use lucide's stroke style (`strokeWidth="2"`, `fill="none"`). Internal icons use fill style. This creates visual inconsistency in demos. Is this acceptable long-term?
4. **Catalog gap closure** — Of the 79 missing icons, how many could be added to the internal catalog to reduce the shim? Some (e.g., `DatabaseIcon`, `CreditCardIcon`, `ExternalLinkIcon`) seem like plausible business-domain additions.
5. **MinusIcon and GripVerticalIcon overlap** — Both appear in `missing-icons.tsx` and in the internal catalog. The shim versions may shadow or conflict with internal versions depending on import resolution. This needs verification.
